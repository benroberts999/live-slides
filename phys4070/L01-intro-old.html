Why computational physics?

// phys4070/7270
  • Many (most) problems in physics cannot be solved exactly, even in theory
  • Many problems are simply too big/complex to solve without computers
  • New experiments: ridiculous amounts of data
    • e.g., LHC: ~ 1 PB ~ 1000 TB per second! (beam on)
    • e.g., LIGO: ~ 10 TB per day (always)

1. Why use C++?

// phys4070/7270
  • Fast/powerful, Predictable resource control
    • "direct" map to hardware: CPU, memory
  • Back-compatible with C - every* C program is a valid C++ program
  • Has a well-defined standard + standard library (<algorithm>, <numeric> etc.)
  • Multi-paradigm (generic, functional, object-oriented, data-o, etc.)
  • Object-oriented: encapsulation
  • High and low level language
    • Propaganda: "High level language that leaves no room for a lower-level one"
    • Propaganda: "Zero-cost abstraction"
    • Write nice high-level OO code, same(/better) performance as C or Fortran
  • ​Directly interface with other languages (C, Fortran etc.) - LAPACK
  • Widely used: many experts, lots of existing programs/libraries etc.

2. Why learn C++?

// phys4070/7270
  • Learning lower-level language gives understanding of many concepts (e.g., memory management)
  • A good understanding of C++ will make learning other languages easier, and give you a deeper understanding of computer science
  • => Help you write better code
  • Widely used and highly valued in computer science and industry

3. Down sides?

// phys4070/7270
  • Huge language, many features
  • Many old, obsolete features (for back-compatibility)
    • Can make it hard to learn
  • Often, lots of bad/out-dated "advice" online
  • Very powerful; can be easy to "shoot yourself in the foot"
    • ​It will happily let you do almost anything.. including very bad ideas
  • Often quite 'verbose' - sometimes requires more code/typing than other languages

Writing good code

// phys4070/7270

Goal:

  • Clear, easily understandable code
  • Easy to maintain, use, debug
    • And mark!

Tools:

  • Naming of variables/functions
  • Logic and structure of program/functions
  • Encapsulation (Wrapping code into functions, classes)
  • Following "best principles", "good practices"
    • Learn from decades of past mistakes; takes time/experience
  • Good commenting practices
  • Enable warnings!: -Wall (also: -Wextra -Wpedantic -Wconversion)
// phys4070/7270
double f(double x, double y) {
  double a, b, c;
  a = (4.0 / 3) * M_PI * x * x * x;
  b = (4.0 / 3) * M_PI * x * y * y;
  c = a - b;
  if (c > 0.0) {
    return c;
  } else {
    return -c;
  }
}

Bad:

  • Uninitialised variables
    • Hard to debug, hard to follow, easy to make mistakes
  • Code transcription; copy/paste errors
    • Extra work, hard to maintain, 2x probability of error
  • Unclear naming
    • Confusing, hard to understand
// phys4070/7270
double sphere_volume(double radius) {
  return (4.0 / 3) * M_PI * std::pow(radius, 3);
}

double volume_difference(double radius1, double radius2) {
  const double volume1 = sphere_volume(radius1);
  const double volume2 = sphere_volume(radius2);
  return std::abs(volume1 - volume2);
}

Good:

  • Variables defined where used
  • Makes use of functions
  • Clear naming
